Create `skills/shared/validation-loop.md` with three sections in this order:

**Section 1: Fix One Thing at a Time**
Generic version of the rule — NO TypeInto/KeyboardShortcut or selector examples. Content to include:
- Core principle: When an error occurs, identify the root cause, fix ONLY that, and re-run
- Never bundle a speculative improvement with the actual fix
- Changing two things at once makes it impossible to verify which change resolved the issue or whether the extra change introduced a new one
- One fix per iteration, re-run, verify

**Section 2: Validation Iteration Loop**
Merge coded's loop (coding-guidelines.md lines 98-108) with RPA's loop (validation-and-fixing.md lines 46-60). Unified procedure:

```
REPEAT:
  1. uip rpa validate --file-path "<FILE>" --project-dir "<PROJECT_DIR>" --format json
  2. IF 0 errors -> EXIT to Smoke Test
  3. Identify highest-priority error
  4. Fix one thing (see rule above)
  5. GOTO 1
```

Include these specific elements:
- Use `validate` command (NOT `get-errors`). Explain: `validate` forces Studio to re-analyze the file AND returns errors; `get-errors` only returns cached state
- Use `--file-path` to target the specific file changed — faster than validating the whole project
- Cap at 5 validation fix attempts — after 5 failed attempts, present remaining errors to the user (they may require domain knowledge or environment-specific fixes)
- 2 runtime retry attempts after smoke test failures — after 2 failed retries, present the user with error details, suggested fix, and options (A/B/C)
- All 4 DO NOT rules from RPA, verbatim:
  1. DO NOT stop until all errors are resolved (or cannot be resolved automatically)
  2. DO NOT obsess on one error — if it cannot be resolved, skip it, continue, and defer to user
  3. DO NOT skip validation steps
  4. DO NOT assume edits worked without checking
- The fix-one-thing rule is the 5th DO NOT (DO NOT bundle multiple fixes)
- Do NOT include `focus-activity` — that is RPA-domain-specific
- Add a brief cross-reference: "See [cli-reference.md](cli-reference.md) for full `validate` and `run-file` command documentation."

**Section 3: Smoke Test**
Lift from validation-and-fixing.md lines 64-91, generalized:
- Explanation: `validate` (static analysis) and `run-file` (runtime compilation) use different validation paths — some errors only surface at runtime
- Command: `uip rpa run-file --file-path "<FILE>" --format json` (use generic `<FILE>` placeholder, no `.xaml` or `.cs` extension)
- Also show with `--input-arguments` variant and `--log-level Verbose` variant
- When to run (3 conditions):
  1. Workflow has no compilation errors but you want to verify runtime behavior
  2. Workflow involves file I/O, API calls, or data transformations that could fail at runtime
  3. User specifically asks to test the workflow
- When NOT to run (3 conditions):
  1. Workflow has side effects (sends emails, modifies databases, calls external APIs) — warn user first
  2. Workflow requires interactive input (UI automation, attended triggers)
  3. Compilation errors still exist (fix those first)
- If runtime errors occur, loop back to fix (links to the iteration loop above)

**Style conventions (match Phase 1 shared files):**
- Paradigm-neutral language throughout — no C#, XAML, ObjectRepository, Descriptors references
- Procedural style with numbered steps and code blocks
- Self-contained — covers the complete topic
