# Bindings Reference

Bindings connect agent features and resources to their corresponding Orchestrator entities (memory space indexes, escalation apps, etc.). For low-code agents, bindings are managed by Studio Web through the `.agent-builder/bindings.json` file.

## How Bindings Work

When an agent uses features (memory spaces) or resources (escalations, tools, contexts, MCPs), each one needs to be bound to an actual Orchestrator resource at deployment time. The binding maps a local reference key to a named resource in the UiPath platform.

```
feature.json (referenceKey) ──► bindings.json ──► Orchestrator Memory Space Index
resource.json (properties)  ──► bindings.json ──► Orchestrator App / Process / Index
```

## bindings.json v2.0 Schema

The bindings file lives at `.agent-builder/bindings.json`. Do not edit this file directly — it is managed by Studio Web.

```json
{
  "version": "2.0",
  "resources": [
    {
      "resource": "<RESOURCE_TYPE>",
      "key": "<RESOURCE_KEY>",
      "value": {
        "name": {
          "defaultValue": "<RESOURCE_NAME>",
          "isExpression": false,
          "displayName": "<DISPLAY_LABEL>"
        }
      },
      "metadata": {
        "bindingsVersion": "2.2",
        "solutionsSupport": "true"
      }
    }
  ]
}
```

## Resource Types

| `resource` Value | Bound To | Used By |
|-----------------|----------|---------|
| `app` | Action Center escalation app | Escalation resources |
| `memorySpace` | Orchestrator memory space index | Memory space features |
| `tool` | Process / API workflow / Agent | Tool resources |
| `context` | ECS index | Context resources |
| `mcp` | MCP server | MCP resources |

## Binding Examples

### Escalation App Binding

Links an escalation resource to its Action Center app:

```json
{
  "resource": "app",
  "key": "COE28595_EscalationApp",
  "value": {
    "name": {
      "defaultValue": "COE28595_EscalationApp",
      "isExpression": false,
      "displayName": "App name"
    }
  },
  "metadata": {
    "bindingsVersion": "2.2",
    "solutionsSupport": "true"
  }
}
```

The `key` matches the `properties.appName` in the escalation resource's `resource.json`.

### Memory Space Binding

Links a memory space feature to its Orchestrator index:

```json
{
  "resource": "memorySpace",
  "key": "MemorySpace",
  "value": {
    "name": {
      "defaultValue": "MemorySpace",
      "isExpression": false,
      "displayName": "Memory name"
    }
  },
  "metadata": {
    "bindingsVersion": "2.2",
    "solutionsSupport": "true"
  }
}
```

The `key` matches the `memorySpaceName` in the feature's `feature.json`.

## Relationship to Features and Resources

### Memory Space → Binding

```
features/<Name>/feature.json
  └── memorySpaceName: "MemorySpace"
        │
        ▼
.agent-builder/bindings.json
  └── resources[]: { resource: "memorySpace", key: "MemorySpace" }
```

### Escalation → Binding

```
resources/<Name>/resource.json
  └── channels[].properties.appName: "COE28595_EscalationApp"
        │
        ▼
.agent-builder/bindings.json
  └── resources[]: { resource: "app", key: "COE28595_EscalationApp" }
```

## Empty Bindings

An agent with no features or resources has an empty bindings file:

```json
{
  "version": "2.0",
  "resources": []
}
```

## Important Notes

1. **Do not edit `.agent-builder/bindings.json` directly** — Studio Web manages this file. When you add a feature or resource to the project, Studio Web creates the corresponding binding entry.

2. **Binding resolution happens at deployment** — the binding's `defaultValue` must match an actual resource name in the target Orchestrator folder.

3. **Each feature/resource needs exactly one binding** — missing bindings cause deployment failures. If you add a resource locally, it will need to be bound through Studio Web before deployment.

4. **The `key` field is the link** — it connects the feature/resource configuration to the binding entry. Ensure consistency between the resource's reference name and the binding key.
